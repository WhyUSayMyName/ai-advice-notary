# CLAUDE.md — контекст проекта «Якорь» (ai-advice-notary)

Система доказуемой фиксации неизменности цифровых данных: документы и телеметрия
хранятся off-chain у владельца, во внешний реестр (блокчейн) якорится только
SHA-256-хеш. Внешний аудитор постфактум доказывает подмену или уничтожение данных,
**не доверяя оператору хранилища** — это центральное свойство, ради которого
принимаются все архитектурные решения. Система обнаруживает нарушения, а не
предотвращает их. Формализованная модель угроз и план экспериментов — в docx-архиве
исследования (вне репозитория); публичный план — в `ROADMAP.md`.

## Структура репозитория

| Путь | Что это |
|---|---|
| `contracts/Notary.sol` | Единственный контракт: append-only реестр `notarize(hash)` + `anchorRoot(root, leafCount)` |
| `scripts/deploy-notary.ts` | Деплой; пишет `NOTARY_ADDRESS` в оба `.env` и экспортирует ABI в приложение |
| `test/` | Тесты контракта (hardhat) |
| `blockchain_notary/` | Electron + React десктоп-приложение |
| `blockchain_notary/src/main/*-core.ts` | **Ядра без Electron** — вся логика живёт здесь |
| `blockchain_notary/src/mcp/` | MCP-сервер (нотаризация LLM-диалогов и аттестаций) |
| `verifier-cli/` | Независимый верификатор аудитора — отдельный мир, см. ниже |

## Архитектурные принципы (не нарушать)

1. **Ядра свободны от Electron.** `database-core`, `anchor-service`, `merkle-core`,
   `evidence-core`, `dialog-canon`, `attestation-canon`, `canonical-json` не импортируют
   `electron`. Electron-модули (`database.ts`, `anchor.ts`, `version-chain.ts`) — тонкие
   обёртки. Новая логика добавляется в ядро с DI (см. `ChainAdapter`, инъекция `now()`),
   обёртка только связывает. Это делает всё тестируемым в vitest и переносимым
   в будущий headless-сервис (ROADMAP, phase 7).

2. **Верификатор не должен доверять этому репозиторию.** `verifier-cli/verify.mjs` —
   один читаемый файл, одна зависимость (`ethers`), без SQLite и без общего кода
   с приложением. Аудитор читает его глазами. Не выноси в него общие модули,
   не добавляй зависимости, не усложняй.

3. **On-chain минимализм.** В контракт уходят только `bytes32` (хеш или merkle-root).
   Никаких URI, метаданных, персональных данных — это и приватность, и газ.
   Проверка merkle-пруфов — off-chain (в верификаторе), НЕ в контракте.

4. **Детерминизм канонов — святое.** Любая неоднозначность ломает воспроизводимость
   проверки:
   - `canonical-json.ts`: сортировка ключей, запрет NaN/Date/функций (ошибка, не искажение);
   - merkle: `leaf = SHA-256(0x00 ‖ h)`, `node = SHA-256(0x01 ‖ min ‖ max)` (доменное
     разделение против second-preimage, сортированные пары — пруф без флагов лево/право),
     нечётный узел поднимается без пары (не дублируется — CVE-2012-2459), всюду SHA-256;
   - форматы версионированы: `notary-dialog/v1`, `notary-attestation/v1`,
     `notary-evidence/v2`. Изменение формата = новая версия + обратная совместимость,
     старые форматы никогда не перестают проверяться.

5. **Anchor-воркер последовательный.** Один in-flight запрос — нет гонок nonce.
   Перед КАЖДОЙ отправкой — `isNotarized` (дубликаты on-chain невозможны).
   Recovery при старте сверяет незавершённые записи с чейном (и по хешу, и по
   root'ам батчей). Масштабирование — через merkle-батчи, не через параллелизм.

6. **Очередь в SQLite — источник истины**, переживает падения. База общая между
   Electron-приложением и MCP-сервером → WAL + busy_timeout уже включены
   в `createDatabase`, не отключать.

## Инварианты данных

- Хеш файла: `0x` + 64 hex, SHA-256 **от байтов файла** (`sha256FileHex`).
  Канонизированный диалог/аттестация СТАНОВИТСЯ файлом — поэтому весь конвейер
  (очередь, батчи, evidence, аудит, верификатор) работает с ним без спецслучаев.
  Сохраняй это свойство: новый тип артефакта = новый канон → файл, а не новая ветка кода.
- БД: `UNIQUE(artifact_id, hash)` — одинаковый контент в разных артефактах разрешён,
  внутри одного — конфликт (`HashConflictError` со `conflictVersion`) либо `unchanged`,
  если совпал с последней версией. Молчаливых `INSERT OR IGNORE` в проекте нет — не возвращать.
- Версии связаны `previous_hash`; целостность цепочек проверяет `version-chain-core`.
- ABI контракта: единственный источник — `blockchain_notary/src/main/abi/Notary.json`,
  генерируется `deploy-notary.ts` из hardhat-артефакта. Не редактировать руками,
  не хардкодить строки ABI в коде.

## Сборка и тесты

```shell
npm install && cd blockchain_notary && npm install

# всё сразу: hardhat node + деплой + приложение
cd blockchain_notary && npm run dev:all

npx hardhat test                      # контракт (из корня)
cd blockchain_notary && npm test      # vitest-ядра
npx tsc                               # типы приложения (noEmit)
npm run build:mcp                     # MCP-сервер → dist-mcp/server.mjs
```

Живые e2e (нужен поднятый узел + деплой):
`E2E_RPC_URL=... E2E_NOTARY_ADDRESS=... E2E_PK=... npm test`

**Ловушка better-sqlite3:** основная копия собрана под ABI Electron. Всё, что
работает в чистом Node (vitest, MCP-бандл), использует алиас `better-sqlite3-node`
(vitest.config.ts / scripts/build-mcp.mjs). Не «чинить» это npm rebuild'ом —
сломаешь либо приложение, либо тесты.

**MCP-сервер — stdio:** stdout принадлежит JSON-RPC. Любой вывод — только в stderr
(баннер esbuild уже перенаправляет console.log). Не добавлять console.log в код,
попадающий в бандл сервера.

## Конвенции

- Комментарии в коде приложения — по-русски; комментарий объясняет ограничение
  или намерение, а не пересказывает строку.
- Тесты рядом с кодом в `__tests__/`; для ядер — обязательны. Тесты шагают через
  `processNext()` и инъецированные часы — таймеров и sleep'ов в тестах нет.
- `.env` не коммитится (в git-истории его никогда не было — сохранять это).
  Шаблон — `blockchain_notary/.env.example`.
- Коммиты: `feat:`/`chore:`/`docs:` + содержательное тело; пуш в `main` на
  github.com/WhyUSayMyName/ai-advice-notary.
- Дизайн UI утверждён: язык Linear (тёмный графит, индиго #5E6AD2, обе темы) —
  макет в артефакте, стиль переносится в React-компоненты при декомпозиции App.tsx.
